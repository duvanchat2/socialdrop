# PR #13 — 🔴 Auditoría de endpoints sin isolación por userId

**Área:** Seguridad
**Prioridad:** 🔴 Crítico
**Esfuerzo estimado:** L (1-2 días)

**Problema:**
Varios servicios operan por `id` sin verificar pertenencia: `posts.findOne/update/remove/retry`, `queue.remove`, `brain.markPublished`, `flows`/`sequences` por id, `metrics.deleteGoal`. Un id filtrado permite leer/mutar recursos de otro usuario.

**Solución propuesta:**
Añadir `where: { id, userId }` (o verificación explícita de ownership con 404 si no coincide) en todas las lecturas/mutaciones por id. Tras PR #12, el `userId` viene del token. Añadir tests de acceso cross-user por módulo.

**Archivos a modificar:**
- `apps/api/src/modules/posts/posts.service.ts`
- `apps/api/src/modules/queue/queue.service.ts`
- `apps/api/src/modules/brain/brain.service.ts`
- `apps/api/src/modules/flows/flows.service.ts`
- `apps/api/src/modules/sequences/sequences.service.ts`
- `apps/api/src/modules/metrics/metrics.service.ts`

**Criterios de aceptación:**
- [ ] Toda mutación por id valida ownership
- [ ] Test cross-user devuelve 404/403 en cada módulo
