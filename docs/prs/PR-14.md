# PR #14 — Relaciones FK + índices multi-tenant en schema; IG por profileId

**Área:** Feature / Performance
**Prioridad:** 🟡 Importante
**Esfuerzo estimado:** L (1-2 días)

**Problema:**
Muchos modelos llevan `userId` suelto **sin FK** a `User` (`ContentItem`, `Flow`, `Contact`, `YoutubeComment`, `YoutubeAutoReply`, `Competitor`, `GeneratedScript`, `PlatformMetrics`, `PostAnalytics`), varios con `@default("demo-user")` → sin integridad referencial ni cascade delete. Además `InstagramProvider.post` usa `INSTAGRAM_ACCOUNT_ID` de **env global** (una sola cuenta para toda la plataforma) — incompatible con multi-usuario/multi-cuenta. Lo mismo aplica a `FACEBOOK_PAGE_ID`.

**Solución propuesta:**
1. Añadir `user User @relation(..., onDelete: Cascade)` + `@@index([userId, ...])` a todos los modelos con `userId` suelto; quitar los `@default("demo-user")`.
2. Publicación IG/FB: leer el id de cuenta desde `Integration.profileId` (por integración) en vez de env. Pasar `profileId` en `PostContent` desde el processor.
3. Migración con backfill de `demo-user` antes de imponer las FK.

**Archivos a modificar:**
- `libs/prisma/prisma/schema.prisma` (+ migración con backfill)
- `libs/integrations/src/providers/instagram.provider.ts`
- `libs/integrations/src/providers/facebook.provider.ts`
- `apps/api/src/modules/posts/processors/post-scheduler.processor.ts`

**Criterios de aceptación:**
- [ ] Borrar un `User` cascada a todos sus datos
- [ ] Publicar en IG usa el `profileId` de la integración (multi-cuenta posible)
- [ ] Migración corre sobre datos existentes sin pérdida
